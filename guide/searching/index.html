    Performing Searches¶

    This guide explains how to search for person records using the Tetra Connect Search Service.

    Overview¶

    The Search Service provides matching capabilities for finding person records based on name and date of birth. It offers fast, scalable searching with configurable search options.

    Search Endpoint¶

    Basic Search¶

    GET /api/search?firstName=John&lastName=Smith&dob=1980-01-15

    Required Parameters:

    • firstName - First name to search for
    • lastName - Last name to search for
    • dob - Date of birth in ISO format (yyyy-MM-dd)

    Optional Parameters:

    • middleNames - Middle names (defaults to empty if not provided)
    • maxNoOfResults - Maximum number of results to return
    • minimumResponseScore - Minimum score threshold (defaults to 0.6)

    Example Request¶

    curl "http://localhost:1967/api/search?\
firstName=John&\
lastName=Smith&\
dob=1980-01-15&\
maxNoOfResults=10&\
minimumResponseScore=0.7"

    Search Response¶

    Response Structure¶

    {
  "personSearch": {
    "firstName": "John",
    "lastName": "Smith",
    "middleNames": "",
    "dob": "1980-01-15"
  },
  "searchResults": [
    {
      "externalId": "12345",
      "firstName": "John",
      "middleNames": "Michael",
      "lastName": "Smith",
      "dob": "1980-01-15",
      "calculatedScore": {
        "type": "WEIGHTED",
        "name": "Person Score",
        "context": "PERSON",
        "value": "0.95",
        "weight": 100,
        "childScores": [...]
      }
    }
  ],
  "noOfResults": 1,
  "totalNumberOfRecordsSearched": 1000000,
  "searchTimeInMillis": 45,
  "scoreTimeInMillis": 120,
  "totalTimeInMillis": 165
}

    Response Fields¶

    Top Level:

    • personSearch - Echo of the search parameters
    • searchResults - Array of matching results, ranked by score
    • noOfResults - Number of results returned
    • totalNumberOfRecordsSearched - Total documents in the index
    • searchTimeInMillis - Time taken for search phase
    • scoreTimeInMillis - Time taken for scoring phase
    • totalTimeInMillis - Total processing time

    Search Result:

    • externalId - External identifier from source data
    • firstName - First name
    • middleNames - Middle names (may be empty)
    • lastName - Last name
    • dob - Date of birth
    • calculatedScore - Detailed scoring breakdown

    Score-Based Ranking¶

    All search results are scored based on relevance using multiple strategies:

    • String Distance
    • Phonetic Similarity
    • Date Range Matching
    • Synonym Matching

    Results are ranked by their calculated score, with the highest scoring matches returned first.

    Filtering Results¶

    Score Threshold¶

    Use minimumResponseScore to filter out low-quality matches:

    # Only return results with score >= 0.8
curl "http://localhost:1967/api/search?\
firstName=John&lastName=Smith&dob=1980-01-15&\
minimumResponseScore=0.8"

    Limiting Results¶

    Use maxNoOfResults to limit the number of returned results:

    # Return only top 5 matches
curl "http://localhost:1967/api/search?\
firstName=John&lastName=Smith&dob=1980-01-15&\
maxNoOfResults=5"

    Performance Recommendation

    Setting maxNoOfResults can improve performance for queries that would otherwise return many results. The Score Service will stop processing once the limit is reached.

    Response Formats¶

    Tetra Connect supports two response formats:

    Protocol Buffers¶

    Protocol Buffers is the primary response format, providing optimal performance through a compact binary format:

    curl -H "Accept: application/x-protobuf" \
  "http://localhost:1967/api/search?..."

    Java Client Integration:

    Tetra Connect provides Protocol Buffers model API JARs that you can use in your Java client application:

    // Add the dependency (published to GitHub Packages)
// uk.co.tetra.connect:search-api-model:version

import uk.co.tetra.connect.search.proto.ProtoSearchRequest;
import uk.co.tetra.connect.search.proto.ProtoTetraConnectResult;

// Use the models in your client code
ProtoTetraConnectResult result = searchServiceClient.search(request);

    The Protocol Buffers models are available as JAR artefacts, allowing you to write strongly-typed Java clients without manually parsing responses.

    JSON¶

    JSON is also supported for easier debugging:

    curl -H "Accept: application/json" \
  "http://localhost:1967/api/search?..."

    Performance Recommendation

    For production integrations, use Protocol Buffers with the provided Java model JARs. Protocol Buffers are faster to serialise/deserialise and consume less bandwidth than JSON.

    Checking Index Size¶

    To check how many records are currently in the search index:

    curl "http://localhost:1967/api/size"

    Response: 

    1000000

    This returns the total number of indexed records.

    Tuning Search and Scoring Behaviour¶

    Tetra Connect provides extensive configuration options via environment variables to tune search and scoring behaviour to match your data quality and performance requirements.

    Service Configuration

    Configuration is split between two services:

    • Search Service - Controls query matching behaviour (fuzzy matching, date ranges for filtering)
    • Score Service - Controls scoring behaviour (field weights, phonetic/synonym scaling)

    Field Weighting (Score Service)¶

    Control how much each field contributes to the final score:

    Setting Environment Variable Default Description
    First Name Weight TETRA_CONNECT_FIELD_WEIGHT_FIRST_NAME 30 Weight for first name in scoring
    Middle Names Weight TETRA_CONNECT_FIELD_WEIGHT_MIDDLE_NAMES 10 Weight for middle names in scoring
    Last Name Weight TETRA_CONNECT_FIELD_WEIGHT_LAST_NAME 40 Weight for last name in scoring (highest by default)
    Date of Birth Weight TETRA_CONNECT_FIELD_WEIGHT_DOB 30 Weight for date of birth in scoring

    The weights determine relative importance. For example, with the default configuration, last name has the highest impact on the final score.

    Fuzzy Matching Tolerance (Search Service)¶

    Configure how lenient fuzzy matching is for each field. Each field (firstName, lastName, dob) supports the following settings:

    First Name Fuzzy Matching¶

    Setting Environment Variable Default Description
    Maximum Edit Distance TETRA_CONNECT_QUERY_FIRSTNAME_MAXEDITS 2 Maximum edit distance (0 = exact match only, 1-2 = fuzzy, allows 0-2 character differences)
    Prefix Match Length TETRA_CONNECT_QUERY_FIRSTNAME_PREFIXLENGTH 0 Number of initial characters that must match exactly
    Allow Transpositions TETRA_CONNECT_QUERY_FIRSTNAME_TRANSPOSITIONS true Whether to allow character transpositions (e.g., "Jhon" "John")
    Query Expansion Limit TETRA_CONNECT_QUERY_FIRSTNAME_MAXEXPANSIONS 50 Limits query expansion (higher = more CPU, more matches)

    Last Name Fuzzy Matching¶

    Setting Environment Variable Default Description
    Maximum Edit Distance TETRA_CONNECT_QUERY_LASTNAME_MAXEDITS 2 Maximum edit distance (0 = exact match only, 1-2 = fuzzy, allows 0-2 character differences)
    Prefix Match Length TETRA_CONNECT_QUERY_LASTNAME_PREFIXLENGTH 0 Number of initial characters that must match exactly
    Allow Transpositions TETRA_CONNECT_QUERY_LASTNAME_TRANSPOSITIONS true Whether to allow character transpositions (e.g., "Jhon" "John")
    Query Expansion Limit TETRA_CONNECT_QUERY_LASTNAME_MAXEXPANSIONS 50 Limits query expansion (higher = more CPU, more matches)

    Date of Birth Fuzzy Matching¶

    Setting Environment Variable Default Description
    Maximum Edit Distance TETRA_CONNECT_QUERY_DOB_MAXEDITS 2 Maximum edit distance (0 = exact match only, 1-2 = fuzzy, allows 0-2 character differences)
    Prefix Match Length TETRA_CONNECT_QUERY_DOB_PREFIXLENGTH 0 Number of initial characters that must match exactly
    Allow Transpositions TETRA_CONNECT_QUERY_DOB_TRANSPOSITIONS true Whether to allow character transpositions (e.g., "Jhon" "John")
    Query Expansion Limit TETRA_CONNECT_QUERY_DOB_MAXEXPANSIONS 50 Limits query expansion (higher = more CPU, more matches)

    Date of Birth Matching Range (Both Services)¶

    Configure the tolerance for date of birth matching:

    Setting Environment Variable Default Description
    DOB Tolerance Range TETRA_CONNECT_DATE_OF_BIRTH_RANGE_IN_DAYS 5 Tolerance for DOB matching (5 days). Dates outside this range receive a score of 0.

    Dates within the range are scored based on proximity to the search date.

    DOB Range Impact

    With RANGE_IN_DAYS=5:

    • Search: 1980-01-15
    • Match: 1980-01-15 Score: 1.0 (exact match)
    • Match: 1980-01-17 Score: ~0.6 (2 days difference, within range)
    • Match: 1980-01-20 Score: ~0.0 (5 days difference, at boundary)
    • Match: 1980-01-22 Score: 0.0 (7 days difference, outside range)

    Configure on Both Services

    This setting must be configured on both Search Service and Score Service to ensure consistent behavior. The Search Service uses it for initial date filtering, while the Score Service uses it for scoring date proximity.

    Phonetic and Synonym Scaling (Score Service)¶

    Configure scoring for phonetic and synonym matches:

    Setting Environment Variable Default Description
    Phonetic Match Scaling TETRA_CONNECT_SCALE_PHONETICS 0.9 Score multiplier for phonetic matches (90% of exact match score)
    Synonym Match Scaling TETRA_CONNECT_SCALE_SYNONYMS 0.9 Score multiplier for synonym matches (90% of exact match score)

    These values scale down the score when a match is found via phonetics or synonyms rather than exact/fuzzy matching.

    Score Threshold (Search Service)¶

    Set the minimum score for results to be returned:

    Setting Environment Variable Default Description
    Minimum Response Score TETRA_CONNECT_DEFAULT_MINIMUM_RESPONSE_SCORE 0.6 Minimum score threshold for returning results. Can be overridden per request via minimumResponseScore query parameter.

    Results with a score below this threshold are filtered out.

    Custom Synonyms (Both Services)¶

    Add custom first name synonyms using environment variables:

    Setting Environment Variable Format Example Description
    First Name Synonyms TETRA_CONNECT_SYNONYMS_FIRST_NAMES_{NAME} TETRA_CONNECT_SYNONYMS_FIRST_NAMES_WILLIAM=bill,billy,will Define synonyms for first names. {NAME} is the primary name (uppercase), value is comma-separated synonyms (lowercase).

    Additional Examples:

    TETRA_CONNECT_SYNONYMS_FIRST_NAMES_ROBERT=bob,rob,robbie
TETRA_CONNECT_SYNONYMS_FIRST_NAMES_JAMES=jim,jimmy
TETRA_CONNECT_SYNONYMS_FIRST_NAMES_ELIZABETH=liz,beth,betty

    When a search uses "William", results containing "Bill", "Billy", or "Will" will be matched with a score scaled by TETRA_CONNECT_SCALE_SYNONYMS (default: 0.9).

    Configure on Both Services

    Custom synonyms must be configured on both Search Service and Score Service for proper functionality:

    • Search Service - Uses synonyms during query matching to find records
    • Score Service - Uses synonyms during scoring to apply synonym scaling factor

    Configuration Summary¶

    The following table summarises which service manages each configuration setting:

    Configuration Search Service Score Service Notes
    Field Weighting (TETRA_CONNECT_FIELD_WEIGHT_*) Controls relative importance of fields in scoring
    Fuzzy Matching (TETRA_CONNECT_QUERY_*) Controls query matching tolerance
    DOB Range (TETRA_CONNECT_DATE_OF_BIRTH_RANGE_IN_DAYS) Configure on both services
    Phonetic/Synonym Scaling (TETRA_CONNECT_SCALE_*) Score multipliers for phonetic/synonym matches
    Score Threshold (TETRA_CONNECT_DEFAULT_MINIMUM_RESPONSE_SCORE) Filters results below threshold
    Custom Synonyms (TETRA_CONNECT_SYNONYMS_FIRST_NAMES_*) Configure on both services
